Continuous batching from scratch

How iteration-level scheduling makes LLM serving fast — built by hand, then measured: static 10 steps / 65% utilization → continuous 6 steps / 100%, same requests served.

Balaji Chidambaram · June 2026 · ~6 min read

I've worked with LLM APIs a lot over the years, but I've always been fascinated by how everything works under the hood. Since undergrad I've been drawn to GPUs and distributed systems — one of my projects was large-scale processing with Hadoop and MapReduce — but I never had the resources to go deep, since access to serious GPUs was expensive back in 2020 (it's even more expensive now 🤣).

Working with LLM APIs day-to-day now — Vertex AI, OpenAI, Claude — that old curiosity came back: how do these services serve live, millisecond inference for massive models at scale? I want to pivot my career toward inference and platform engineering, and this blog — starting with nanogpt-served — is me learning it by building it from scratch.

When a server runs one model for many users, the requests arrive and finish at different times. The hard question isn't which output belongs to which request (that's solved by batch row) — it's when do you decide the batch? That single timing choice is the gap between a slow server and a fast one.

01Static batching — the slow default

Grab a batch, run it in lockstep, everyone leaves when the slowest finishes. Five requests needing [2, 6, 3, 4, 2] tokens, 3 GPU lanes:

request decoded (useful) idle lane — finished request, dead weight no lane (batch smaller than max)
Two wastes: idle lanes (R1 done at step 2 but still decoded through step 6) and the wave barrier — R4/R5 can't start until all of wave 1 drains, even though lanes sat free from step 3.

02Continuous batching — re-decide every step

vLLM's term is iteration-level scheduling: the scheduler runs between every decode step. A finished request leaves immediately; a queued one takes the freed lane next step.

request decoded (useful) no lane (batch shrank — not wasted)
Watch lane A: R1 (blue) finishes at step 2, R4 (purple) takes its lane at step 3 — no barrier, no wait. The only gap (lane C, step 6) is the unavoidable drain tail: the batch shrinks to 2, it does not decode dead weight. That's why util reads 100%, not 94%.

03The scoreboard

A slot-step = one GPU lane used for one decode step — the atomic unit of work. total = lanes you paid for; useful = lanes that produced a token a request still wanted. Same model, same prompts, same 5 requests served:

modestepsuseful / total
slot-steps		utilizationserved
static1017 / 2665%5
continuous617 / 17100%5
Identical work, identical output. Continuous batching didn't skip anything — served and useful are unchanged. It just stopped wasting lanes: 26→17 slot-steps (no idle lanes) and 10→6 steps (no head-of-line wait). On a real GPU those two numbers are throughput ↑ and time-to-first-token ↓.
Why useful == total in continuous? You evict at the end of every step, so at decode time every lane holds a request that still wants a token. No lane is ever spent on a finished request. Static can't evict mid-wave (it's lockstep), so the gap is pure waste.

04Policy vs mechanism — the vLLM shape

The build accidentally drew the real architecture boundary. The scheduler decides batch membership each iteration (policy); the decode just runs a forward pass (mechanism) and is oblivious to which requests it holds.

This is vLLM in miniature: a Scheduler over a model-execution layer. The missing piece between this toy and vLLM is inside the mechanism box.

05The core decision: admit → decode → evict

The scaffolding (request queue, the decode pass) is plumbing. The whole idea lives in the ordering of the step loop — admit waiting requests into free lanes, decode the current batch once, then evict whoever finished so their lane opens up next step:

# ADMIT: fill every free lane (while, not if).
while len(self.running) < self.max_batch_size and self.waiting:
    self.running.append(self.waiting.pop(0))

# DECODE: one forward over the current batch.
self._decode(self.running)
self.steps += 1
self.total_slot_steps  += len(self.running)   # cost
self.useful_slot_steps += len(self.running)   # all useful → evict-at-end

# EVICT: finished requests leave, freeing their lane for next step.
done = [r for r in self.running if r.done]
self.finished.extend(done)
self.running = [r for r in self.running if not r.done]

06What's next — why a KV cache doesn't just drop in

The deliberate cheat: _decode recomputes the full context every step — no KV cache. A naive single-sequence KV cache assumes one sequence of one length; a batch whose membership and lengths change every step shatters that single contiguous tensor. That exact mismatch is what PagedAttention solves — a cache manager that hands out fixed-size blocks so ragged, churning sequences share GPU memory without copying. That's the next build.

Building this drove home something I hadn't appreciated: inference isn't only about the GPU — it's just as much about the framework and software layered above it, from tensors up to the scheduler. The batch scheduler packages requests together so the GPU's parallel compute actually gets used instead of sitting idle. LLM inference used to be a black box to me — I always wondered why inference APIs charge less per token for batch requests and cached prompts. Now I understand the machinery behind it.
